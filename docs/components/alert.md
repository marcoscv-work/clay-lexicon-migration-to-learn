# Alert

Use an alert to tell the user about the status of a task or the system, such as a
success, a warning, or an error. Give the alert a short type label and a clear
message. Alerts are intrusive by design, so use them for information the user
needs to notice.

## Variants and anatomy

### Types

Four status types communicate the nature of the message through color and icon.
Try each one live in the [Alert story](https://storybook.clayui.com/?path=/story/design-system-components-alert--default),
switching the display type in the controls.

- **Information** (blue): relevant information shown while the user works.
- **Success** (green): the action completed successfully.
- **Warning** (yellow): the action completed, but something needs attention.
- **Error** (red): something went wrong and the action did not complete.

### Anatomy

An alert has four parts: a status icon, the type text, the description text, and
an optional close button. Set the status icon and the type text in semibold so
the message type is easy to scan. An alert can span multiple lines and can
contain links, which use the primary color.

### Variations

- **Feedback indicator**: the smallest form of an alert, a status icon used to
  draw attention in place, for example inside a form or a sidebar. If the icon's
  meaning is not clear from nearby text, show a tooltip on hover. Try the
  [feedback story](https://storybook.clayui.com/?path=/story/design-system-components-alert--feedback).
- **Toast**: a temporary notification that does not interrupt the user. Toasts
  appear at the bottom left of the screen and stack vertically. Try the
  [toast story](https://storybook.clayui.com/?path=/story/design-system-components-alert--toast).
- **Embedded**: an alert placed inside a form. Its width follows the container,
  and the close action is usually not needed.
- **Stripe**: a full width bar, a space saving alternative to embedded and toast
  alerts, placed near form sections or headers.
- **Inline action**: an action placed after the message on the same line, instead
  of on its own line. Use it for messages of up to two lines, and set a maximum
  width so alerts stay consistent.

## Usage guidelines

- Keep temporary alerts on screen long enough to read: about 5 seconds with no
  action, and about 10 seconds when the alert has an action. Freeze the countdown
  while the pointer is over the alert, and resume it on mouse out.
- Keep one primary action per alert, so the user knows the main thing to do.
- Represent actions with buttons and navigation with links.
- Use the information type for most embedded alerts in forms.
- Do not use an alert for content the user does not need to act on or notice; use
  quieter feedback instead.

## Implementation

Install the component and the CSS package:

```shell
npm install @clayui/css @clayui/alert
```

Set the status with `displayType`, which accepts `info`, `success`, `warning`,
`danger`, and `secondary`:

```jsx
import Alert from '@clayui/alert';

export default function Example() {
	return (
		<Alert displayType="success" title="Success">
			The item was created.
		</Alert>
	);
}
```

Choose a form with the `variant` property (`feedback`, `inline`, or `stripe`),
and add actions with the `actions` property:

```jsx
import Alert from '@clayui/alert';
import Button from '@clayui/button';

export default function Example() {
	return (
		<Alert
			actions={<Button alert>View</Button>}
			displayType="info"
			title="Heads up"
			variant="stripe"
		>
			A new version is available.
		</Alert>
	);
}
```

For temporary notifications, render alerts inside `Alert.ToastContainer` and give
each alert an `autoClose` time in milliseconds:

```jsx
import Alert from '@clayui/alert';

export default function Example({items, onClose}) {
	return (
		<Alert.ToastContainer>
			{items.map((value) => (
				<Alert autoClose={5000} key={value} onClose={() => onClose(value)} title="Saved">
					Item {value} was saved.
				</Alert>
			))}
		</Alert.ToastContainer>
	);
}
```

## Accessibility

- By default an alert renders with an ARIA `alert` role so assistive technology
  announces it. Set `role` deliberately when you place many static alerts on a
  page, so screen readers are not flooded.
- Give the alert a `title` that names the status ("Success", "Error"), so the
  meaning does not depend on color alone.
- Toasts freeze their auto close timer on hover and focus, which also gives
  keyboard and screen reader users time to reach any action inside the alert.
